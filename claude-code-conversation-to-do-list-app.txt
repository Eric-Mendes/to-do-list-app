     Plan: To-Do List App

     Context

     Build a single-user to-do list app from scratch using Streamlit, SQLite,
     and Docker. The project directory is currently empty. The goal is a
     polished, modern app with list/task CRUD, tag-based folder grouping, a
     calendar/timeline view, and an analytics dashboard — all delivered
     incrementally as 8 PRs (40 commits), each independently verifiable and
     covered by tests.

     ---
     Architecture

     Stack

     - UI: Streamlit >= 1.36 (with st.navigation multi-page API)
     - DB: SQLite via raw sqlite3 (no ORM — single user, minimal deps)
     - Calendar: streamlit-calendar (FullCalendar wrapper)
     - Charts: Plotly Express + pandas
     - Container: Docker + docker-compose
     - CI: GitHub Actions (ruff lint, black format check, pytest --cov, docker
     smoke test)

     Directory Structure

     to-do-list-app/
     ├── .github/workflows/ci.yaml
     ├── .streamlit/config.toml          # dark theme
     ├── app/
     │   ├── main.py                     # st.navigation entrypoint
     │   ├── database/
     │   │   ├── connection.py           # get_connection(), reads DB_PATH env
     var
     │   │   └── migrations.py          # run_migrations(conn) — CREATE TABLE IF
      NOT EXISTS
     │   ├── models/
     │   │   ├── list_model.py           # TaskList CRUD + get_lists_by_tag
     │   │   ├── task_model.py           # Task CRUD + complete_task + analytics
      queries
     │   │   └── tag_model.py            # Tag CRUD + assign/remove from list
     │   ├── pages/
     │   │   ├── home.py                 # List grid with tag filter
     │   │   ├── list_detail.py          # Tasks inside a list
     │   │   ├── calendar_view.py        # Timeline (streamlit-calendar)
     │   │   └── analytics.py            # KPI metrics + Plotly charts
     │   ├── components/
     │   │   ├── sidebar.py              # Tag tree + nav
     │   │   ├── task_card.py            # Checkbox, priority badge, due date
     chip
     │   │   ├── list_card.py            # List name, task counts, overdue badge
     │   │   ├── modals.py               # @st.dialog create/edit/delete dialogs
     │   │   └── charts.py               # Plotly wrappers with app palette
     │   └── utils/
     │       ├── date_helpers.py         # is_overdue, days_until_due,
     format_due_date_label
     │       └── styles.py               # CSS injection
     ├── tests/
     │   ├── conftest.py                 # in-memory SQLite fixture
     │   ├── unit/                       # per-model + date helper tests
     │   └── integration/                # multi-model workflow tests
     ├── Dockerfile
     ├── docker-compose.yml
     ├── requirements.txt
     └── requirements-dev.txt

     Database Schema

     tags         (id, name UNIQUE, color DEFAULT '#6366f1', created_at)
     task_lists   (id, name, description, created_at, updated_at)
     list_tags    (list_id FK→task_lists, tag_id FK→tags, PRIMARY
     KEY(list_id,tag_id))
     tasks        (id, list_id FK→task_lists, title, description,
                   status
     CHECK('pending','in_progress','completed','cancelled'),
                   priority CHECK('low','medium','high'),
                   due_date, completed_at, created_at, updated_at)
     Foreign keys have ON DELETE CASCADE. Dates stored as ISO-8601 strings.

     ---
     Key Decisions

     - Raw sqlite3 over SQLAlchemy: minimal deps, readable code, fits
     single-user scale
     - Tag-as-folder: lists can belong to multiple tags simultaneously; more
     flexible than a strict hierarchy
     - @st.cache_resource for DB: connection created once per server process,
     safe for single-user SQLite
     - In-memory DB for tests: no file I/O, fast, isolated per test function via
      conftest.py fixture
     - Dark theme from the start: cleaner default than Streamlit's light theme;
     colors anchored in config.toml

● Fixed. The root cause: st.markdown() processes content through the Markdown
  parser first — any lines with 4+ spaces of leading indentation are treated as
  code blocks and rendered as literal text. The indented f-strings in our
  components all had 12+ spaces of padding, triggering this.

   st.html(), which bypasses the Markdown parser entirely and renders the HTML
  directly.
